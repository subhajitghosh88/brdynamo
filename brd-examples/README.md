# BRD Examples

This folder contains sample Business Requirements Documents (BRDs) in multiple formats to test the BRDynamo application.

## Available Examples

### 1. E-commerce Product Management System
- **Files**: `e-commerce-product-management.txt`, `e-commerce-product-management.docx`, `e-commerce-product-management.pdf`
- **Description**: Comprehensive BRD for an e-commerce product catalog management system
- **Features**: Product CRUD, category management, inventory tracking, React/Flask tech stack
- **Complexity**: High - Full enterprise application with detailed requirements

### 2. Minimal Product API
- **Files**: `minimal-product-api.txt`, `minimal-product-api.docx`, `minimal-product-api.pdf`
- **Description**: Simple BRD for basic product API testing
- **Features**: Basic CRUD operations for products with minimal requirements
- **Complexity**: Low - Perfect for quick testing and validation

### 3. Task Management System
- **Files**: `task-management-system.txt`, `task-management-system.docx`, `task-management-system.pdf`
- **Description**: Comprehensive collaborative task management platform
- **Features**: Project management, Kanban boards, real-time collaboration, GraphQL API
- **Complexity**: High - Modern full-stack application with advanced features

### 4. Library Management System
- **Files**: `library-management-system.txt`, `library-management-system.docx`, `library-management-system.pdf`
- **Description**: Digital library management platform for book lending
- **Features**: Book catalog, member management, lending operations, reporting
- **Complexity**: Medium - Traditional CRUD application with business logic

## Usage Instructions

1. **Testing Different Formats**: Upload any of the `.txt`, `.docx`, or `.pdf` versions to test BRDynamo's parsing capabilities across different file formats.

2. **Complexity Testing**: 
   - Use `minimal-product-api.*` for quick testing and validation
   - Use `library-management-system.*` for medium complexity scenarios
   - Use `e-commerce-product-management.*` or `task-management-system.*` for comprehensive testing

3. **Format Comparison**: Upload the same BRD in different formats (TXT, DOCX, PDF) to verify consistent parsing results.

## File Formats

- **TXT**: Plain text format, easily readable and editable
- **DOCX**: Microsoft Word format with formatted headings and structure
- **PDF**: Portable Document Format for cross-platform compatibility

## Notes

- All examples follow standard BRD structure with sections like:
  - Project Overview
  - Stakeholders
  - Functional Requirements
  - Technical Specifications
  - Data Models
  - API Endpoints (where applicable)
  - Security Requirements
  - Timeline and Success Criteria

- The examples cover different technology stacks and architectural patterns to test BRDynamo's versatility in code generation.
